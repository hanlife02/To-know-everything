# 小红书内容监控、AI 理解与 Bark 推送项目规划

## 1. 项目目标

构建一个本地运行、以安全和隐私为优先的内容监控系统，针对小红书上的指定第三方账号：

- 抓取账号发布的最新笔记和视频内容
- 提取帖子正文、图片文字、视频语音内容和高赞评论
- 使用 AI 进行翻译、摘要和视频内容理解
- 将新增内容通过 Bark 推送
- 提供本地管理后台，用于配置账号、Cookie、模型、任务和历史记录

项目约束：

- 平台第一阶段只做 `小红书`
- 第一阶段只验证 `1 个账号`
- 抓取优先考虑 `官方 API`，如不可行再评估 `低风控的登录态/cookie 方案`
- 不接受高风控方案
- 不云部署
- 不保存原始视频文件
- 安全第一，尽量降低封号风险

## 2. 当前已确认需求

### 2.1 监控对象

- 监控目标是 `其他人发布的公开内容`
- 第一阶段只支持 `1 个小红书账号`
- 后续架构需要保留扩展到多账号的能力

### 2.2 抓取范围

单条内容需要尽可能覆盖以下信息：

- 帖子正文
- 标题
- 发布时间
- 链接
- 图片中的文字 OCR
- 视频语音转写结果
- 视频“在讲什么”的理解结果
- 评论区高赞评论

评论抓取规则：

- 仅抓取高赞评论
- 默认取前 `N` 条
- `N` 由后台配置

### 2.3 AI 处理范围

每条内容进入 AI 流程后，至少生成：

- 简体中文译文
- 中文摘要
- 视频内容理解结果
- 适用于 Bark 的短推送文案

说明：

- 视频如果没有现成字幕，走外部模型 API 转写
- 图片文字通过 OCR 提取
- 视频文件本身不保存，只保存转写结果和理解结果

### 2.4 推送规则

支持两种触发方式：

1. 手动推送
2. 定时推送

手动推送定义：

- 立即抓取目标账号最新内容
- 推送自上次推送以来的所有新增内容

定时推送定义：

- 到达计划时间后执行抓取
- 仅推送自上次推送以来的增量内容

频率要求：

- 不需要高频轮询
- 以“推送前抓取一次”为主
- 调度支持自定义 `每 X 小时 / 每天 / 每周`

### 2.5 推送渠道

- MVP 第一阶段只做 `Bark`
- 推送渠道需要可开关
- 后续保留扩展 Telegram Bot 的接口设计

### 2.6 模型接入方式

模型平台要求：

- 仅支持 `OpenAI 兼容接口`
- 配置项至少包含 `base_url`、`api_key`
- 支持先获取 `/models` 列表，再由用户选择模型
- 支持导入多条模型配置
- 支持在不同任务上选择不同模型
- 允许发送给第三方模型接口
- OCR、视频转写、翻译、摘要、理解允许分别使用不同的 OpenAI 兼容服务商

建议拆分的模型用途：

- 摘要模型
- 翻译模型
- OCR 模型
- 视频转写模型
- 视频理解模型

### 2.7 数据保留策略

需要保存：

- 正文
- 评论
- OCR 文本
- 转写文本
- 译文
- 摘要
- 视频理解结果

不保存：

- 原始视频文件

保留策略：

- 默认长期保存
- 后台支持按时间范围删除历史记录

### 2.8 安全要求

- 不云部署，仅本地运行
- 不泄露隐私信息
- Cookie 与 API Key 尽量本地加密存储
- 如果加密存储实现不合适，则降级为运行时输入
- 项目设计目标是尽量降低封号风险
- 前端必须提供鉴权能力，不能作为无保护的本地面板直接暴露

## 3. MVP 范围

第一阶段只验证“小红书单账号 -> AI 处理 -> Bark 推送 -> 后台可管理”这条主链路。

MVP 包含：

- 单账号配置
- Cookie 配置与连通性测试
- 手动抓取
- 手动推送新增内容
- 定时推送增量内容
- 帖子正文抓取
- 图片 OCR
- 视频转写
- 高赞评论抓取
- 中文翻译
- 中文摘要
- 视频内容理解
- Bark 推送
- 历史记录展示与删除
- 多模型配置管理

MVP 不强求：

- 多平台支持
- 多账号并发抓取
- Telegram Bot
- 云端部署
- 复杂权限系统
- 原始媒体文件存档

## 4. 核心功能拆分

### 4.1 抓取层

职责：

- 拉取指定小红书账号的最新内容
- 标准化输出统一数据结构
- 处理去重、失败重试、限频
- 尽量复用稳定、低风险的请求链路

抓取链路建议分层：

1. 账号内容列表抓取
2. 单条内容详情抓取
3. 评论抓取
4. 媒体文本提取调度

说明：

- 若无法获得官方接口，则评估低频、低并发的 Cookie 会话访问
- 不建议把高频浏览器自动化作为 MVP 主方案

统一内容模型建议：

```json
{
  "platform": "xiaohongshu",
  "account_id": "xxx",
  "account_name": "Some Creator",
  "content_id": "unique-id",
  "content_type": "post_or_video",
  "title": "note title",
  "body_text": "post body",
  "ocr_text": "text extracted from images",
  "transcript_text": "speech transcript",
  "video_understanding": "what the video is about",
  "top_comments": [
    {
      "author": "commenter",
      "like_count": 123,
      "text": "comment text"
    }
  ],
  "url": "https://...",
  "published_at": "2026-03-25T10:00:00Z",
  "metrics": {},
  "raw_payload": {}
}
```

### 4.2 媒体文本提取层

职责：

- 对图片执行 OCR
- 对视频音频执行转写
- 将多模态内容统一整理为可供摘要的文本

处理结果建议：

- `ocr_text`
- `transcript_text`
- `merged_source_text`

### 4.3 AI 处理层

职责：

- 清洗文本
- 判断是否值得推送
- 生成简体中文译文
- 生成摘要、关键观点和标签
- 生成视频理解结果
- 针对 Bark 生成简短推送文案

建议处理顺序：

1. 整理帖子正文、OCR 文本、转写文本、评论文本
2. 如原文不是中文，生成简体中文译文
3. 生成标准摘要
4. 若是视频内容，额外生成“视频在讲什么”的理解结果
5. 生成 Bark 推送短文案

建议输出结构：

```json
{
  "translated_text": "简体中文译文",
  "summary_short": "一句话摘要",
  "summary_medium": "3 到 5 条要点",
  "video_understanding": "这个视频主要在讲什么",
  "keywords": ["AI", "产品", "发布"],
  "importance_score": 0.88,
  "push_text": "适合 Bark 的最终消息"
}
```

### 4.4 推送层

职责：

- 发送 Bark 推送
- 记录推送日志
- 避免重复发送
- 支持手动触发与定时触发

推送行为规则：

- 推送单位为“新增内容”
- 每次推送前执行抓取
- 根据上次成功推送记录决定增量范围

### 4.5 任务调度层

职责：

- 定时执行抓取任务
- 将新内容放入 AI 处理流程
- 异常报警和重试
- 支持按小时、每天、每周的调度配置

### 4.6 后台管理层

职责：

- 管理监控账号
- 管理 Cookie
- 管理模型配置
- 管理 Bark 配置
- 提供前端登录鉴权
- 查看内容历史与推送日志
- 手动执行抓取和推送
- 删除指定时间范围的数据

## 5. 建议技术架构

推荐技术栈：

- 后端语言：Rust
- 后端框架：Axum
- 异步运行时：Tokio
- 数据库：SQLite 起步，后续可迁移到 PostgreSQL
- 查询层：SQLx
- 调度：Tokio 任务 + Cron 表达式调度库
- 加密：本地密钥派生 + 对称加密
- 抓取：HTTP 客户端 + Cookie 会话管理
- 前端：React + Vite
- AI：OpenAI 兼容接口
- 部署：本地 Docker Compose 或本机直接运行

鉴权建议：

- MVP 采用本地单用户登录
- 使用用户名 + 密码登录
- 密码仅保存哈希
- 会话使用本地签名 Token 或安全 Cookie

版本策略：

- 优先使用当前最新稳定版
- 初始化项目时按当时的最新稳定版本落地

## 7. 数据流设计

整体流程建议如下：

1. 用户在前端配置监控账号、Cookie、模型与 Bark
2. 用户手动触发，或调度器到点触发任务
3. 系统先抓取目标账号最新内容列表
4. 根据 `platform + account_id + content_id` 去重
5. 对新增内容抓详情、评论和媒体文本
6. 组织原始文本并调用模型完成转写、翻译、理解和摘要
7. 生成 Bark 推送文案
8. 发送增量推送
9. 保存内容、摘要、译文和推送日志

## 8. 平台接入策略

建议原则：

- 第一优先：官方 API 或官方开放能力
- 第二优先：低频、低并发、登录态的受控访问
- 第三优先：如果实现路径明显高风控，则暂不做

针对小红书的实现要求：

- 只抓公开可见内容
- 严格限制访问频率
- 避免多线程高并发抓取
- 账号 Cookie 只在本地使用
- 将抓取器做成可替换模块，便于后续调整策略

## 9. AI 流程设计

建议把 AI 能力拆成独立步骤，避免一个超大提示词承担所有任务：

1. OCR
2. 视频转写
3. 文本合并
4. 翻译为简体中文
5. 内容摘要
6. 视频理解
7. Bark 推送文案生成

这样做的原因：

- 便于为不同任务选不同模型
- 出错时更容易重试
- 后续更容易统计成本和质量

建议后台允许以下模型映射：

- `ocr_model`
- `transcription_model`
- `translation_model`
- `summary_model`
- `understanding_model`

## 10. Bark 推送设计

适合：

- 即时提醒
- 简短摘要
- 单条新增内容通知

建议消息格式：

```text
[小红书] 某账号有新内容
标题：...
摘要：...
评论亮点：...
链接：...
```

如有多条新增内容，建议：

- 默认逐条推送
- 后续可加“本次新增汇总模式”

默认目标：

- 默认使用 `iPhone Bark App` 的标准推送地址
- 后续再扩展自托管 Bark Server

## 12. 安全设计

敏感信息包括：

- Cookie
- API Key
- Bark 密钥或服务地址

建议方案：

- 优先使用本地加密存储
- 加密主密钥来自用户输入的主密码或本机安全存储
- 前端不直接明文持久保存敏感字段
- 后端日志默认脱敏
- 导出功能默认不包含敏感配置

如果本地安全存储在 MVP 中实现复杂，可降级为：

- 敏感字段不落盘
- 服务启动后由用户录入
- 进程退出即失效

## 14. 风险与注意事项

### 14.1 平台风险

- 小红书对自动化访问、接口调用和登录态使用存在不确定性
- 如果只能依赖脆弱的非公开接口，维护成本会较高
- 高风控实现路径与当前目标冲突，应避免
- 如果无法找到可接受的低风控抓取方案，则项目第一阶段不能进入“假抓取器占位”路线，必须先解决真实抓取可行性

### 14.2 安全风险

- Cookie 一旦泄露可能影响账号安全
- 第三方模型接口会接触抓取文本，需要由用户自行配置并承担外发风险

### 14.3 成本风险

- OCR、视频转写、摘要、翻译都可能单独计费
- 若后续扩展到多账号，成本会快速上升

### 14.4 质量风险

- OCR 可能不准
- 评论排序和高赞定义可能受接口限制
- 视频理解质量依赖转写质量

## 16. 当前仍待确认的问题

当前关键约束已经基本确认，剩余问题从“需求决策”变成“实现验证”：

1. 小红书是否存在可接受的低风控抓取方案，需要在实施前优先验证
2. 前端本地鉴权采用何种最终形态，需要在实现时在 `账号密码 + 本地会话` 方案上细化
3. 本地敏感信息加密存储是采用主密码解锁，还是退化为运行时输入，需要在安全实现阶段定稿
4. 小红书评论和媒体提取接口能力是否满足需求，需要通过实际 PoC 验证
